# WebAssembly SPA Оболочка: Wasm Frontend (wasm_frontend.md)

Пользовательский интерфейс приложения реализован в виде статической SPA-страницы (Single Page Application), работающей полностью в браузере пользователя без необходимости внешнего сервера базы данных или бэкенд-вычислений.

---

## 1. Компоненты интерфейса и файлы

Фронтенд состоит из следующих файлов в корневом каталоге:
* **[index.html](file:///g:/Мой%20диск/SNR_Lib/ADC_Model/index.html)** — структура страницы, блоки ввода параметров, контейнеры для графиков и логика моста Python-JS.
* **[index.css](file:///g:/Мой%20диск/SNR_Lib/ADC_Model/index.css)** — строгая инженерная дизайн-система с нейтральной светлой темой по умолчанию, дополнительной темной темой, адаптивной сеткой, компактными контролами и индикатором загрузки движка Wasm.

---

## 2. Архитектура интеграции WebAssembly и PyScript

Интеграция выполнена через **PyScript (версия 2024.1.1)** на базе интерпретатора **Pyodide**:
- В конфигурации PyScript указана загрузка только математических библиотек `numpy` и `scipy`:
  ```json
  {
      "packages": ["numpy", "scipy"],
      "files": {
          "src/solver.py": "./src/solver.py",
          "src/analyzer.py": "./src/analyzer.py",
          "src/capacitors.json": "./src/capacitors.json"
      }
  }
  ```
- Код автоматически копирует файлы `solver.py`, `analyzer.py` и общий каталог `capacitors.json` с сервера в локальную файловую систему Pyodide, обеспечивая их импорт через `from src.solver import ...`.

## 3. Мост взаимодействия Python -> JavaScript -> Plotly.js

Для достижения мгновенного запуска приложения мы отказались от установки Python-пакета `plotly` в виртуальном окружении Wasm (что сэкономило ~15 секунд при каждой загрузке страницы). Вместо этого рендеринг графиков делегирован легковесной JS-библиотеке **Plotly.js**, загружаемой через CDN.

### Схема взаимодействия:
1. Пользователь меняет параметры на HTML-форме (вкладка *Time Domain*).
2. Событие перехватывается прокси-обработчиком Python:
   ```python
   change_proxy = create_proxy(run_single_simulation)
   js.document.getElementById("r_ext").addEventListener("input", change_proxy)
   ```
3. Python-решатель рассчитывает массивы данных симуляции во временной области и спектральные метрики.
4. Данные конвертируются из `numpy.ndarray` в стандартные списки Python (`.tolist()`) и передаются в JavaScript-функцию `renderPlotlyChart`:
   ```python
    js.renderPlotlyChart("time-plot", to_js_obj(time_data), to_js_obj(time_layout))
    ```
5. Функция `Plotly.react` обновляет график «на лету» без пересоздания DOM-узлов, обеспечивая плавность при перетаскивании ползунков.

Для интерактивных полей используется debounce-задержка перед запуском одиночного сценария. Это снижает нагрузку на Pyodide при быстром перемещении слайдеров и сохраняет ручной запуск для тяжелого многопараметрического свипа.

Справочник компонентов загружается JavaScript-слоем через `fetch('./src/capacitors.json')`, а Python-слоем из того же файла внутри виртуальной файловой системы Pyodide. Каноническая единица емкости в каталоге — фарад; UI переводит значение в нФ только для отображения.

В верхней панели интерфейса отображается состояние расчета (`Ready`, `Updating...`, `Sweep running...`) и длительность последнего завершенного расчета. Для одиночного режима PyScript передает в решатель `trace_until`, чтобы в браузер попадал только видимый участок временной диаграммы; спектральные метрики продолжают рассчитываться по полному массиву выборок.

Подпись временного графика рассчитывается по фактически доступному окну трассы. Если запрошенные `4 / f_in` длиннее полной длительности массива выборок `N_samples / f_s`, интерфейс не пишет «первые 4 периода», а показывает реальную длительность в миллисекундах и долю периода. Это особенно важно для низких частот входного сигнала, где четыре периода могут быть существенно длиннее текущего FFT-окна.

Во вкладке `Design Space` диапазоны осей sweep задаются пользователем через `min`, `max`, количество точек и тип сетки (`Linear`/`Log`). Значения \(T_{acq}\) вводятся в наносекундах и переводятся в секунды перед передачей в Python-ядро. Для защиты интерактивного режима запрещены одинаковые оси X/Y, логарифмические диапазоны с неположительными границами и сетки более 400 точек. Результат представлен одной полноширинной тепловой картой ENOB Loss; график параллельных координат удален как дублирующий данные и менее читаемый при двух входных осях.

Для объяснения потери ENOB в одиночном сценарии добавлены два интерфейсных слоя:
- диагностическая панель `Dominant limiter`, где сравниваются идеальный SINAD, измеренный SINAD, потеря динамического диапазона в дБ и пиковый residual в LSB/мВ;
- график `ENOB Error View`, показывающий residual после вычитания лучшей основной гармоники из восстановленных выборок АЦП, а также отдельную кривую ошибки установления.

Панель предупреждений различает клиппинг и ограничения спектральной записи. Сообщение `SPECTRAL LIMIT` появляется, если в 1024 отсчетах меньше четырех периодов алиасированной частоты, вход алиасится в DC, находится в пределах одного FFT-бина от Найквиста или выбран несинусоидальный тестовый сигнал. В этих режимах числа остаются доступными, но не должны интерпретироваться как достоверная оценка ENOB синусоидального АЦП-теста.

В первой вкладке также добавлена единая таблица `Single Point Numeric Results`. Она объединяет все числовые результаты одной рабочей точки: основной ENOB/SINAD/THD результат, residual/settling и ablation-диагностику. Для ablation одна рабочая точка пересчитывается в нескольких диагностических режимах: линейный baseline с идеальным измерением \(V_{ext}\), режим `C(V) only` с идеальным измерением \(V_{ext}\), режим `Settling only` на линейном конденсаторе и полный combined-режим. Первая строка таблицы является основной метрикой `Total ENOB loss`, а разность между combined и суммой отдельных вкладов выводится как `Interaction`. Ошибка установления показывается как аналоговое напряжение с LSB-эквивалентом.

Интерфейс использует строгую визуальную иерархию без декоративных градиентов, стеклянных эффектов, свечения и объемных теней. Рабочие блоки имеют малые радиусы, нейтральные поверхности и единый синий цвет только для действий и активного состояния. Карточка High SNR Channel и рекламные места сохранены, но оформлены как вторичные служебные блоки, не конкурирующие с расчетными результатами. Таблица результатов не расходует отдельную колонку на повторяющееся имя группы: `Main result`, `Spectrum and residual` и `Loss decomposition` разделены компактными секционными строками и слабыми оттенками фона. На мобильном экране таблица сокращается до наиболее важных столбцов без горизонтального переполнения.

Графики используют общую приглушенную инженерную палитру: серо-синий, бирюзовый, пыльно-фиолетовый, охристый и приглушенный красный. Чистые насыщенные цвета исключены; различимость поддерживается толщиной линий, штриховкой и маркерами. Тепловая карта сохраняет смысловой переход от малой к большой потере ENOB, но использует спокойную последовательность от светлого серо-зеленого к охристо-красному.

Во вкладке `Methodology` собраны схема моделируемого объекта, определения сигналов, уравнения \(C(V)\), settling, SINAD/ENOB, THD и residual, описание ablation-декомпозиции, численного метода и границ применимости. Электрическая схема выполнена как встроенный SVG с принятыми схемотехническими обозначениями источника, резисторов, ключа, конденсаторов, узлов и земли; она масштабируется без потери четкости. При открытии этой вкладки панель входных параметров скрывается, чтобы справочный материал использовал всю ширину окна. Названия числовых показателей в `Single Point Numeric Results` являются контекстными ссылками и открывают соответствующий раздел методологии. Каноническое описание содержания вкладки приведено в **[model_methodology.md](file:///g:/Мой%20диск/SNR_Lib/ADC_Model/wiki/concepts/model_methodology.md)**.

Во вкладке `Library` панель параметров Single Point скрывается, и каталог получает всю ширину рабочей области. Широкая компоновка использует предсказуемое соотношение колонок таблицы и графика `60/40`, причем обе колонки имеют явное разрешение на сжатие внутри CSS Grid. Ширины столбцов каталога заданы через `colgroup`, поэтому содержимое отдельных компонентов не изменяет геометрию всей вкладки. На мобильной ширине полная техническая таблица сохраняет ширину столбцов и прокручивается только внутри своего контейнера. Сравнительный Plotly-график не строится, пока вкладка скрыта: первый рендер выполняется после двух кадров компоновки открытой вкладки, а дальнейший размер синхронизируется через `ResizeObserver`.

Эта логика реализована в PyScript-части **[index.html](file:///g:/Мой%20диск/SNR_Lib/ADC_Model/index.html)**. Residual-график использует уже рассчитанные массивы `v_rec`, `v_samples`, `v_ext_samples` и `t_samples`, а decomposition-панель добавляет быстрый линейный прогон C0G для baseline/settling-only сравнения. Расчетные определения и ограничения синхронизированы с **[src/solver.py](file:///g:/Мой%20диск/SNR_Lib/ADC_Model/src/solver.py)** и **[src/analyzer.py](file:///g:/Мой%20диск/SNR_Lib/ADC_Model/src/analyzer.py)**.

## 4. Запуск и публикация

### Локальный запуск
Для проверки работы Wasm-приложения требуется локальный веб-сервер (прямое открытие HTML из проводника заблокировано браузерами из-за CORS-политик PyScript).
Запуск сервера из корня проекта:
```bash
python -m http.server 8000
```
После запуска страница доступна по адресу **http://localhost:8000**.

### Публикация
Так как приложение является статическим веб-сайтом, его деплой выполняется простым переносом файлов `index.html`, `index.css` и папки `src/` на **GitHub Pages** (или любой другой статический хостинг).
